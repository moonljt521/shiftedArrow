# 关卡生成与消除判定算法说明

本文档说明箭头消除迷宫游戏的核心算法：棋子模型、消除判定、求解器，以及保证「全覆盖 + 100% 可解」的关卡生成算法。

对应源码：
- `src/gridLogic.ts` — 消除判定 `canEliminate`、求解器 `solve`、死局检测 `hasAnyMove`
- `src/levels.ts` — 关卡生成（哈密顿路径切段 + 反向消除定向）

---

## 1. 棋子模型（Piece）

每个棋子是一条**折线蛇**，由若干个首尾相邻的格子组成：

```
Piece {
  cells: Cell[]   // 有序，cells[0] = 尾部，cells[length-1] = 头部
  dir:   Direction // 头部飞出方向（= 头部那一段的方向）
}
```

关键约束：**`dir` 恒等于头部段方向**，即 `dirBetween(cells[n-2], cells[n-1])`。
这保证了箭头三角贴合在线的末端、并且飞出时是「沿自身折线滑出」而非整体平移。

网格用 `grid[row][col]` 存放占据该格的 `pieceId`，空格为 `null`。

---

## 2. 消除判定 `canEliminate`

点击一条蛇后，它沿头部方向 `dir` 飞出，身体顺着头部轨迹跟随滑出。
因此能否飞出**只取决于头部正前方的射线**是否畅通：

```
从头部格出发，沿 dir 逐格前进直到网格边界：
  若某格被「其它」棋子占据  → 受阻，不可消除
  自身格子不算阻挡（身体会跟着移动让出空间）
全程畅通 → 可消除
```

复杂度：O(射线长度) ≤ O(max(rows, cols))。

---

## 3. 单调性：本游戏的核心性质

> **消除任何一个棋子，只会清空格子，绝不会新增障碍。**

由此推出一个极强的性质——**可消除性是单调的**：
一旦某棋子在当前局面可消除，那么在消除其它任意棋子之后，它**仍然**可消除
（它头部射线上的格子只会变得更空，不会凭空冒出障碍）。

这条性质同时支撑了求解器和生成器的正确性。

---

## 4. 求解器 `solve`（贪心即完备）

```
循环：
  扫描所有剩余棋子，把当前可消除的全部消除
  直到某一轮没有任何棋子可消除为止
若全部消除完 → 返回消除顺序（有解）
否则           → 返回 null（死局）
```

**为什么贪心不需要回溯？** 因为单调性保证「现在能消的，以后也能消」。
所以消除顺序不影响最终能否清空：能解则任意贪心顺序都能解，不能解则怎么排都不行。
复杂度约 O(n²)（n 为棋子数），对本游戏规模绰绰有余。

`hasAnyMove` 是其中一步：是否存在任意可消除棋子，用于运行期死局检测与提示按钮禁用。

---

## 5. 关卡生成：全覆盖 + 保证可解

目标有两个硬性要求：

1. **全覆盖**：所有蛇恰好铺满整个网格，不重叠、不留空白格。
2. **可解**：存在一条把所有蛇依次飞出的合法顺序。

### 5.1 全覆盖 ⇔ 哈密顿路径切段

> 「若干条相邻折线蛇恰好铺满网格、互不重叠」
> 等价于
> 「一条遍历每个格子恰好一次的哈密顿路径，被切成若干连续段」。

因此生成全覆盖布局，只需：**求一条哈密顿路径 → 切段**。每段即一条蛇，
由构造天然保证零空格、零重叠。

#### 第 1 步：起手蛇形路径（serpentine）

按行（或列）来回蛇行遍历整张网格，得到一条初始哈密顿路径：

```
行0:  → → → → →
行1:  ← ← ← ← ←
行2:  → → → → →
...
```

随机选择「按行」或「按列」蛇行，增加多样性。

#### 第 2 步：Backbite 随机化（制造迷宫拐弯）

蛇形路径太规整，需要打乱成迷宫感，但又必须**保持仍是哈密顿路径**。
采用 **backbite（折回）** 操作：

```
设当前路径头部为 path[k]
随机取头部的一个网格邻居 u = path[j]
若 j < k-1：
    反转子段 path[j+1 .. k]
    → u 与原头部相连，新的头部变为原 path[j+1]
```

该操作**不改变所覆盖的格子集合**（只是重排访问顺序），所以结果仍是哈密顿路径，
但每做一次就可能新增/改变若干拐弯。重复 O(格子数 × 常数) 次，
即得到一条充满随机拐弯的迷宫式哈密顿路径。偶尔整体 `reverse` 让尾端也被混洗。

#### 第 3 步：切段

把路径切成长度落在 `[minLen, maxLen]` 的连续段（长度随关卡递增），
并处理末尾：避免切出长度为 1 的碎片（合并进相邻段），保证每条蛇至少 2 格。

### 5.2 反向消除定向（保证可解）

切完段只有几何形状，还需给每条蛇指定**飞出方向**且保证整体可解。
利用单调性，采用**反向消除**构造法：

```
工作网格 work = 所有蛇
循环直到所有蛇都已定向：
  收集候选：某条蛇的某个端点作为头部、沿其末段方向的射线上
            不含「其它尚未定向」的蛇（即此刻该蛇可飞出）
  随机挑一个候选，给该蛇定向（dir = 头部段方向），并从 work 中移除
  若某轮无任何候选 → 死锁，本次生成失败
```

**为什么这样保证可解？** 把蛇「被定向并移除」的先后记为顺序 P。
轮到某条蛇被定向时，`work` 中剩下的恰好是「将在它之后才定向」的蛇，
而我们正是要求它的头部射线对这些蛇畅通才定向的。
所以按顺序 P **正向**消除时：轮到该蛇，比它早的都已消除（不在场），
晚的当初已保证不挡路 → 它必可消除。故 P 是一条合法消除顺序，关卡可解。

> 注意：飞出方向被限制为「头部段方向」，是为了视觉一致（箭头贴线、沿路径滑出）。
> 端点有两个、各自有一个末段方向，定向时两端都可尝试。

### 5.3 重试与复核

backbite 后的某些布局可能在定向阶段死锁（返回 null）。
生成器会**更换随机种子重试**（最多数百次），每次成功后再用 `solve()` **独立复核**确实可解，
双保险确保最终关卡一定可玩。

## 6. 难度设计与 JSON 规则

难度**不是**简单地"箭头更多"。本游戏真正的难度来自**消除顺序被约束的程度**——
你起手有多少选择、依赖链有多深。这些是可以**量化测量**的，所以采用
「JSON 定义难度目标 + 生成器测量并筛选」的方案，而非手工硬调参数。

### 6.1 难度评分 `difficultyScore`（0~1，越大越难）

对一个候选局面，用贪心**逐层剥离**测量两个客观指标：

```
每一轮：把当前所有可消除的棋子同时移除，记为一"层"
  layers      = 总层数（依赖链深度）
  firstLayer  = 第 1 层可立即消除的棋子数
```

- **依赖深度** `depthScore = (layers - 1) / (n - 1)`：层数越多，强制的先后依赖越深 → 越难。
- **起手自由度** `freeRatio = firstLayer / n`：起手能动的越少 → 越难。

```
难度 = 0.6 × depthScore + 0.4 × (1 - freeRatio)
```

无解局面记为 1（最难/不可用），生成时会被筛掉。

### 6.2 JSON 规则文件 `src/levelRules.json`

关卡完全由 JSON 描述，**改 JSON 即改关卡，无需动代码**：

```jsonc
{
  "defaults": {              // 所有关卡的默认值，可被单关覆盖
    "lives": 3,
    "lightning": 5,
    "minPieceLen": 2,
    "maxPieceLen": 6,
    "serpentineAxis": "random", // 起手蛇形按行/列/随机
    "backbiteFactor": 6,        // backbite 强度（越大拐弯越多）
    "maxAttempts": 600,         // 生成重试上限
    "difficulty": { "min": 0, "max": 1 }
  },
  "levels": [
    {
      "id": 1,
      "rows": 10, "cols": 7,    // 网格尺寸
      "maxPieceLen": 4,         // 覆盖默认值
      "timeLimit": 180,         // 倒计时秒数
      "difficulty": { "min": 0.0, "max": 0.3 } // 目标难度区间
    }
    // ... 更多关卡
  ]
}
```

字段说明：

| 字段 | 含义 | 缺省 |
|------|------|------|
| `rows` / `cols` | 网格行列数 | 必填 |
| `timeLimit` | 倒计时秒数 | 必填 |
| `minPieceLen` / `maxPieceLen` | 蛇长度范围（切段用） | 取 defaults |
| `serpentineAxis` | 起手蛇形方向 `rows`/`cols`/`random` | 取 defaults |
| `backbiteFactor` | backbite 次数系数（拐弯密度） | 取 defaults |
| `difficulty` | 目标难度区间 `{min,max}` | 取 defaults |
| `maxAttempts` | 命中难度的最大重试次数 | 取 defaults |

### 6.3 生成器如何命中难度

```
buildLevel(rule)：
  合并 defaults 与本关规则 → 生成参数 + 难度目标 [min,max]
  循环最多 maxAttempts 次：
    换种子生成候选 → solve() 验证可解（不可解则丢弃）
    score = difficultyScore(候选)
    若 min ≤ score ≤ max  → 命中，直接采用
    否则记录"评分离区间中点最近"的候选
  循环结束仍未命中 → 采用最接近的候选（保底，绝不失败）
```

这样既保证**全覆盖 + 可解**（见前几节），又让每关难度落在 JSON 指定区间内。

### 6.4 实测难度曲线

8 个内置关卡的实测难度评分（随关卡平滑递增）：

| 关卡 | 网格 | 难度评分 | 目标区间 |
|------|------|---------|---------|
| 1 | 10×7 | 0.26 | [0.0, 0.30] |
| 2 | 11×7 | 0.33 | [0.10, 0.35] |
| 3 | 11×8 | 0.39 | [0.20, 0.45] |
| 4 | 12×8 | 0.53 | [0.30, 0.55] |
| 5 | 12×9 | 0.43 | [0.40, 0.65] |
| 6 | 13×9 | 0.68 | [0.50, 0.75] |
| 7 | 13×10 | 0.62 | [0.60, 0.85] |
| 8 | 14×10 | 0.71 | [0.70, 1.00] |

> 调参建议：想让某关更难，调高其 `difficulty` 区间、加大 `maxPieceLen` 或 `backbiteFactor`、缩短 `timeLimit`；想加新关卡，直接在 `levels` 数组追加一项即可。

---

## 7. 复杂度小结

| 操作 | 复杂度 |
|------|--------|
| `canEliminate` | O(max(rows, cols)) |
| `solve` / `hasAnyMove` | ~O(n²)，n = 棋子数 |
| `difficultyScore` | ~O(n²)（逐层剥离，复用消除判定） |
| 哈密顿路径 + backbite | O(rows × cols × backbiteFactor) |
| 反向消除定向 | ~O(n² × 射线长度) |
| 生成一关（含重试+可解复核+难度筛选） | 重试次数 × 上述之和，实测毫秒级 |

## 8. 测试覆盖（`tests/gridLogic.test.ts`）

- `canEliminate`：头部畅通可消除、被挡不可消除、自身不阻挡、阻挡物消除后恢复
- `solve` / `hasAnyMove`：所有内置关卡可解、互锁局面无解
- 生成结构：相邻性、界内、无重叠
- **全覆盖**：占用格子数 == rows × cols（零空白）
- **方向一致**：每条蛇 `dir` == 头部段方向
- **难度系统**：评分随关卡整体递增、多数关卡命中 JSON 目标区间
- 难度递增：倒计时单调不增且有下限
